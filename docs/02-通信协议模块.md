# 02 - 通信协议模块 📡

> 📌 本文档详解客户端和服务端之间的通信协议，包括消息格式、编码解码、粘包处理、响应路由。
> 这是整个项目的"血管"——所有数据都通过这套协议流动。

---

## 一、通信协议是什么？

💡 **用大白话说**：两个人打电话，得先约定好"我说中文你听中文"、"我先报姓名你再说事"。通信协议就是客户端和服务器之间的"通话规则"。

### 📊 为什么不用 HTTP 而用自定义 TCP 协议？

| 对比项 | 自定义 TCP 协议（本项目） | HTTP 协议 | 为什么选自定义？ |
|--------|--------------------------|-----------|-----------------|
| 头部开销 | 8 字节（type+length） | 数百字节（Header） | 视频场景消息频繁，省带宽 |
| 连接方式 | 长连接，一直保持 | 短连接，每次重新建 | 不用反复握手，延迟低 |
| 双向通信 | 天然支持（TCP 全双工） | 需要轮询或 WebSocket | 服务器可以主动推送 |
| 消息边界 | 自定义帧头明确标记 | Content-Length 标记 | 控制更精确 |
| 调试难度 | 需要专门工具 | 浏览器就能看 | ✅ 但可以用 qDebug 输出 JSON |

## 二、帧格式详解

### 2.1 一条消息长什么样？

```
┌─────────────┬─────────────┬─────────────────────────────┐
│  type (4B)  │ length (4B) │     JSON payload (变长)       │
│  网络字节序  │  网络字节序  │     UTF-8 编码               │
└─────────────┴─────────────┴─────────────────────────────┘
     4 字节        4 字节            length 字节
```

💡 **用大白话说**：就像寄快递——
- **type** = 快递单上的"物品类型"（1=文件，2=食品...），4 字节整数
- **length** = 快递单上的"重量"，告诉你包裹有多大，4 字节整数
- **JSON payload** = 包裹里的实际内容

### 2.2 为什么 type 和 length 要用"网络字节序"？

⚠️ **大端序 vs 小端序**：
- Intel/AMD 的 CPU 用**小端序**（低位字节在前）：数字 `1` 存为 `01 00 00 00`
- 网络传输标准用**大端序**（高位字节在前）：数字 `1` 存为 `00 00 00 01`

如果客户端是小端序，服务器也是小端序，不转换也行。但如果一端是小端序（x86），另一端是大端序（某些 ARM），不转换就会读出乱码。

**所以**：发送前转成大端序（`qToBigEndian`），接收后转回主机序（`qFromBigEndian`），保证跨平台兼容。

### 2.4 服务端响应的 JSON 格式

```json
{
    "code": 200,
    "message": "登录成功",
    "action": "login",
    "data": {
        "token": "abc123",
        "userId": 1
    }
}
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `code` | int | 状态码，200=成功，其他=失败 |
| `message` | string | 人类可读的消息，如"登录成功"、"密码错误" |
| `action` | string | 这个响应对应什么操作，如"login"、"register" |
| `data` | object | 附加数据，不同操作有不同的字段 |

## 三、发送消息（客户端 → 服务器）

### 3.1 sendMessage() — 所有消息的出口

```cpp
void NetworkManager::sendMessage(const NetworkMessage& msg)
{
    if (m_socket->state() != QAbstractSocket::ConnectedState) {
        emit SIG_error("未连接到服务器"); // 没连上就别发了
        return;
    }

    // 【步骤1】把 JSON 字符串转成 UTF-8 字节序列
    QByteArray data = msg.data.toUtf8();
    // msg.data 是 JSON 字符串，如 {"username":"test","password":"a3f2b8..."}
    // toUtf8() 转成字节序列，因为网络传输的是字节，不是字符串

    // 【步骤2】把 type 和 length 转成网络字节序（大端序）
    quint32 netType   = qToBigEndian(static_cast<quint32>(msg.type));
    quint32 netLength = qToBigEndian(static_cast<quint32>(data.size()));
    // qToBigEndian() 把主机字节序转成大端序
    // static_cast<quint32> 确保是无符号 32 位整数
    // 转换后的值可以直接写入网络字节流

    // 【步骤3】拼接完整的消息帧
    QByteArray message;
    message.append(reinterpret_cast<const char*>(&netType),   sizeof(quint32));
    // 写入 4 字节的 type（大端序）
    // reinterpret_cast 把 quint32* 当成 char* 来读取原始字节
    // sizeof(quint32) = 4，只写入 4 个字节

    message.append(reinterpret_cast<const char*>(&netLength), sizeof(quint32));
    // 写入 4 字节的 length（大端序）

    message.append(data);
    // 写入 JSON 数据体（变长，长度就是 length 的值）

    // 【步骤4】发送
    m_socket->write(message);
    // QTcpSocket::write() 把数据写入发送缓冲区
    // 操作系统的 TCP 协议栈会负责把数据发到服务器
    // write() 是非阻塞的 —— 不会等数据真的发完才返回
}
```

### 3.2 发送示例：sendLogin()

```cpp
void NetworkManager::sendLogin(const QString& username, const QString& password)
{
    // 【步骤1】构造 JSON 对象
    QJsonObject json;
    json["username"] = username;                      // 用户名（明文）
    json["password"] = authPasswordHash(password);    // 密码（SHA-256 加盐哈希）
    // ⚠️ 密码不是明文传输！先加密再发

    // 【步骤2】包装成 NetworkMessage
    NetworkMessage msg;
    msg.type = MSG_LOGIN;                             // 消息类型 = 登录
    msg.data = QJsonDocument(json).toJson(QJsonDocument::Compact);
    // QJsonDocument 把 QJsonObject 序列化成 JSON 字符串
    // Compact 模式 = 没有换行和缩进，节省带宽
    // 结果类似：{"username":"test","password":"a3f2b8c1..."}

    // 【步骤3】发送
    sendMessage(msg);                                 // 走统一的发送流程
}
```

**最终发送的字节流**：
```
00 00 00 02 | 00 00 00 2A | {"username":"test","password":"a3f2..."}
  type=2      length=42      JSON 数据（42 字节）
```

## 四、接收消息（服务器 → 客户端）

### 4.1 粘包问题 — TCP 的"坑"

💡 **什么是粘包？** TCP 是"流式"协议，像水管里的水一样连续不断。

比如服务器连续发两条消息：
```
消息1: [type=3][len=20][20字节的JSON]
消息2: [type=3][len=15][15字节的JSON]
```

但客户端的 `readyRead` 信号可能只触发一次，`readAll()` 拿到的数据可能是：
```
[消息1的35字节][消息2的23字节]  ← 粘在一起了
```

也可能只拿到消息1的一半：
```
[消息1的前10字节]  ← 不完整
```

所以接收端必须**自己处理消息边界**——这就是粘包处理。

### 4.2 onReadyRead() — 粘包处理核心

```cpp
void NetworkManager::onReadyRead()
{
    m_buffer.append(m_socket->readAll());
    // 【步骤1】把收到的所有字节追加到缓冲区
    // m_buffer 是 QByteArray，累积所有还没处理的数据
    // readAll() 读取 socket 中当前可用的所有字节

    while (m_buffer.size() >= 8) {
        // 【步骤2】循环处理，直到缓冲区不够一个帧头
        // 帧头 8 字节 = 4(type) + 4(length)
        // 如果缓冲区连 8 字节都不到，说明连帧头都没收全，退出循环等下次

        quint32 length;
        memcpy(&length, m_buffer.constData() + sizeof(quint32), sizeof(quint32));
        // 【步骤3】从缓冲区读取 length 字段
        // constData() + 4 = 跳过前 4 字节的 type，指向 length
        // memcpy 把 4 个字节拷贝到 length 变量
        // 此时 length 是网络字节序（大端序）

        length = qFromBigEndian(length);
        // 【步骤4】大端序转主机序
        // 现在的 length 就是 JSON 体的实际字节数

        if (static_cast<quint32>(m_buffer.size()) >= 8 + length) {
            // 【步骤5】检查是否收全了一条完整消息
            // 完整消息 = 8字节帧头 + length字节JSON体
            // 如果缓冲区数据量 >= 8+length，说明收全了

            NetworkMessage msg = parseMessage(m_buffer.left(8 + length));
            // 【步骤6】解析这条消息
            // left(8+length) 从缓冲区取出前 8+length 个字节
            // 这就是一条完整的消息

            m_buffer.remove(0, 8 + length);
            // 【步骤7】从缓冲区删除已处理的消息
            // remove(0, n) 从位置 0 开始删除 n 个字节

            if (msg.type == MSG_RESPONSE) {
                if (msg.action.isEmpty()) {
                    // action 为空，用 code 兜底判断
                    qDebug() << "响应缺少 action 字段:" << msg.data;
                    if (msg.code == RESP_PASSWORD_ERROR || msg.code == RESP_USER_NOT_FOUND)
                        emit SIG_loginResult(false, msg.data, QString());
                    else if (msg.code == RESP_USER_EXISTS)
                        emit SIG_registerResult(false, msg.data);
                } else {
                    dispatchResponse(msg);  // 正常路由
                }
            }
        } else {
            break;  // 数据不完整，等下次 readyRead 再处理
        }
    }
}
```

### 4.3 粘包处理的流程图

```
收到数据，追加到 m_buffer
       │
       ▼
  缓冲区 ≥ 8 字节？ ──→ 否 → 等下次
       │
       ▼ 是
  读取 length，计算完整消息长度 = 8 + length
       │
       ▼
  缓冲区 ≥ 8+length？ ──→ 否 → 等下次
       │
       ▼ 是
  取出前 8+length 字节 → parseMessage() → dispatchResponse()
       │
       ▼
  从缓冲区删除已处理的数据
       │
       ▼
  回到"缓冲区 ≥ 8 字节？"继续循环
  （可能缓冲区里还有下一条完整消息）
```

## 五、解析消息

### 5.1 parseMessage() — 从字节流到结构体

```cpp
NetworkMessage NetworkManager::parseMessage(const QByteArray& data)
{
    NetworkMessage msg;                  // 创建空的消息结构体

    if (data.size() < 8) {              // 数据太短，连帧头都不完整
        return msg;                     // 返回空消息
    }

    // 【步骤1】解析帧头
    quint32 type;
    quint32 length;
    memcpy(&type,   data.constData(),             sizeof(quint32));
    // 从偏移 0 读取 4 字节 → type
    memcpy(&length, data.constData() + sizeof(quint32), sizeof(quint32));
    // 从偏移 4 读取 4 字节 → length

    type   = qFromBigEndian(type);      // 大端序 → 主机序
    length = qFromBigEndian(length);

    msg.type = static_cast<MsgType>(type);
    // 把 quint32 转成 MsgType 枚举
    // 比如 type=3 → MSG_RESPONSE

    // 【步骤2】解析 JSON 体
    if (data.size() >= static_cast<int>(8 + length)) {
        QByteArray jsonData = data.mid(8, length);
        // mid(8, length) 从偏移 8 开始取 length 个字节
        // 这就是 JSON 数据

        QJsonDocument doc = QJsonDocument::fromJson(jsonData);
        // 把 JSON 字节序列解析成 QJsonDocument

        if (doc.isObject()) {
            QJsonObject obj = doc.object();
            msg.code = static_cast<ResponseCode>(obj["code"].toInt());
            // 读取状态码，如 200、402

            msg.data = obj["message"].toString();
            // 读取消息文本，如"登录成功"

            msg.action = obj["action"].toString();
            // 读取动作标识，如"login"、"register"
            // 这是路由的关键——决定这个响应分发给谁

            if (obj.contains("data") && obj["data"].isObject()) {
                msg.respData = obj["data"].toObject();
                // 读取附加数据对象
                // 如登录成功后的 token、视频列表等
            }
        }
    }

    // 【步骤3】兼容旧版服务器
    normalizeResponseMessage(msg);
    return msg;
}
```

### 5.2 normalizeResponseMessage() — 兼容旧服务器

💡 **问题**：旧版服务器的 `sendResponse` 没有单独的 `action` 字段，而是把 action 拼在 message 里，用 `|` 分隔，如 `"登录成功|login"`。

```cpp
static void normalizeResponseMessage(NetworkMessage &msg)
{
    // 【步骤1】如果 action 为空，且 message 里有 |，尝试从中提取 action
    if (msg.action.isEmpty() && msg.data.contains(QLatin1Char('|'))) {
        const int sep = msg.data.lastIndexOf(QLatin1Char('|'));
        // 找到最后一个 | 的位置

        const QString tail = msg.data.mid(sep + 1).trimmed();
        // 取 | 后面的部分，如 "login"

        static const QStringList knownActions = {
            "login", "register", "upload_start", "upload_chunk",
            "upload_end", "upload_resume", "video_list", "recommend_feed",
            // ... 所有可能的 action 值
        };

        if (knownActions.contains(tail)) {
            msg.action = tail;                     // 提取出的尾巴就是 action
            msg.data = msg.data.left(sep);         // message 只保留 | 前面的部分
        }
        // 💡 为什么要用 knownActions 验证？
        // 因为 message 本身可能含有 |，如 "上传成功|hls_failed"
        // 只有 | 后面是已知的 action 名，才认为是分隔符
    }

    // 【步骤2】如果还是空，用 code 兜底
    if (!msg.action.isEmpty())
        return;

    if (msg.code == RESP_PASSWORD_ERROR || msg.code == RESP_USER_NOT_FOUND)
        msg.action = "login";            // 密码错误/用户不存在 → 登录场景
    else if (msg.code == RESP_USER_EXISTS)
        msg.action = "register";         // 用户名已存在 → 注册场景
    else if (msg.code == RESP_SUCCESS && msg.respData.contains("token"))
        msg.action = "login";            // 成功且有 token → 登录场景
}
```

## 六、响应路由（dispatchResponse）

### 🎬 就像快递分拣中心：根据地址标签把包裹送到对应的接收人

```cpp
void NetworkManager::dispatchResponse(const NetworkMessage& msg)
{
    const bool success = (msg.code == RESP_SUCCESS);  // 是否成功
    const QString action = msg.action;                // 动作标识
    const QString cleanData = msg.data;               // 消息文本

    // 以下是一个大的 if-else 链，根据 action 把响应分发到不同的信号

    if (action == "register") {
        emit SIG_registerResult(success, cleanData);
        // 注册结果 → RegisterDialog 的 slot_registerResult 接收

    } else if (action == "login") {
        QString token;
        if (success && msg.respData.contains("token"))
            token = msg.respData["token"].toString();  // 提取 token
        m_isLoggedIn = success;
        m_token = success ? token : QString();
        emit SIG_loginResult(success, cleanData, token);
        // 登录结果 → LoginDialog 的 slot_loginResult 接收

    } else if (action == "video_list") {
        QJsonArray videos;
        if (success && msg.respData.contains("videos"))
            videos = msg.respData["videos"].toArray();
        quint64 requestId = 0;
        if (!m_videoListRequestQueue.isEmpty())
            requestId = m_videoListRequestQueue.dequeue();
        emit SIG_videoListResult(success, msg.data, videos, requestId);
        // 视频列表 → VideoMainWindow 接收
        // requestId 用于匹配请求和响应（异步场景下防止乱序）

    } else if (action == "recommend_feed") {
        // ... 类似 video_list

    } else if (action == "upload_start") {
        QString uploadId;
        if (msg.respData.contains("uploadId"))
            uploadId = msg.respData["uploadId"].toVariant().toString();
        if (success)
            m_currentUploadId = uploadId;  // 记录 uploadId，后续上传块要用
        emit SIG_uploadStartResult(success, cleanData, uploadId);

    }
    // ... 其他 action 类似
}
```

## 七、NetworkMessage 结构体

```cpp
struct NetworkMessage {
    MsgType type;           // 消息类型（MSG_LOGIN、MSG_RESPONSE 等）
    QString data;           // 消息内容或响应的 message 字段
    QString action;         // 响应的动作标识（"login"、"register" 等）
    ResponseCode code;      // 响应状态码（200、402 等）
    QJsonObject respData;   // 响应的 data 字段（附加数据）
};
```

💡 **注意**：`data` 字段在不同方向含义不同：
- **发送时**：`data` 存的是 JSON 字符串（整个 payload）
- **接收时**：`data` 存的是解析后的 `message` 字段值

## 八、请求序列号机制

### 为什么需要序列号？

网络请求是异步的。用户快速下滑刷新两次，发了两次 `sendGetVideoList`，两次响应可能乱序到达。没有序列号的话，第一次请求的响应可能被当成第二次的。

```cpp
quint64 NetworkManager::nextListRequestId(QQueue<quint64> &queue)
{
    const quint64 id = ++m_requestSerial;  // 自增序列号
    queue.enqueue(id);                      // 入队（FIFO）
    return id;
}

// 使用示例：
quint64 NetworkManager::sendGetVideoList(int offset, int limit)
{
    // ... 构造消息 ...

    const quint64 requestId = nextListRequestId(m_videoListRequestQueue);
    // 生成一个序列号，加入视频列表请求队列

    sendMessage(msg);
    return requestId;   // 返回给调用者，用于匹配响应
}
```

**匹配机制**：dispatchResponse 中按 FIFO 出队，保证请求和响应顺序对应：
```cpp
quint64 requestId = 0;
if (!m_videoListRequestQueue.isEmpty())
    requestId = m_videoListRequestQueue.dequeue();  // 按入队顺序取出
emit SIG_videoListResult(success, msg.data, videos, requestId);
```

## 九、常见问题

### Q: 为什么所有响应都是 MSG_RESPONSE(type=3)，而不是每种操作一个响应类型？
> 简化设计。所有响应共用一个类型，通过 `action` 字段区分。优点：代码统一，扩展新操作只需加 action；缺点：比每种操作独立类型稍微多一步解析。

### Q: 如果 JSON 特别大，会不会一次 readAll 拿不全？
> 不会。`onReadyRead` 的 while 循环会持续处理：如果缓冲区里不够一条完整消息，就 break 等下次 readyRead 信号。下次追加新数据后继续从帧头解析。

### Q: 为什么用 JSON 而不是 Protobuf？
> JSON 可读性好，调试时 qDebug 直接看内容。Protobuf 二进制格式更省带宽，但开发调试不方便。当前消息量不大，JSON 的开销可以接受。
