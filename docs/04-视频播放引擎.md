## 一、播放引擎整体架构

## 二、VideoState 结构体 — 播放器的"大脑"

### 🎬 就像电视台的导播间：所有设备和状态都在这里

```cpp
typedef struct VideoState {
    // ===== 格式上下文 =====
    AVFormatContext *pFormatCtx;      // FFmpeg 的核心上下文
                                      // 包含视频文件的所有信息
                                      // 如编码格式、时长、流信息等

    // ===== 音频相关 =====
    AVStream *audio_st;               // 音频流对象
    PacketQueue *audioq;              // 音频包队列（生产者:demux, 消费者:audio_callback）
    AVCodecContext *pAudioCodecCtx;   // 音频解码器上下文
    int audioStream;                  // 音频流索引（-1 = 没有音频流）
    std::atomic<int64_t> audio_clock_us{0};  // ⚠️ 音频时钟（微秒）
                                              // atomic = 原子变量，多线程读写安全
                                              // 这是音视频同步的"主时钟"
    SDL_AudioDeviceID audioID;        // SDL 音频设备 ID
    uint8_t audio_buf[288000];        // 音频解码缓冲区
    unsigned int audio_buf_size = 0;  // 缓冲区中有效数据大小
    unsigned int audio_buf_index = 0; // 当前读取位置
    AVFrame *audioFrame;              // 音频帧
    struct SwrContext *swr_ctx;       // 音频重采样上下文
                                      // 把音频转成 SDL 要求的格式（如 S16, 44100Hz, 立体声）

    // ===== 视频相关 =====
    AVStream *video_st;               // 视频流对象
    PacketQueue *videoq;              // 视频包队列（生产者:demux, 消费者:video_thread）
    AVCodecContext *pCodecCtx;        // 视频解码器上下文
    int videoStream;                  // 视频流索引（-1 = 没有视频流）
    std::atomic<int64_t> video_clock_us{0};  // 视频时钟（微秒）
    SDL_Thread *video_tid;            // 视频线程句柄

    // ===== 状态标志 =====
    std::atomic<bool> isPause{false};          // 是否暂停
    std::atomic<bool> quit{false};             // 是否退出
    std::atomic<bool> readFinished{false};     // 是否读完了所有数据
    std::atomic<bool> readThreadFinished{false}; // demux 线程是否结束
    std::atomic<bool> videoThreadFinished{true}; // video 线程是否结束

    // ===== seek 相关 =====
    std::atomic<int> seek_req{0};              // seek 请求标志（0=无请求, 1=有请求）
    std::atomic<int64_t> seek_pos_us{0};       // seek 目标位置（微秒）
    std::atomic<int> seek_flag_audio{0};       // 音频 seek 完成标志
    std::atomic<int> seek_flag_video{0};       // 视频 seek 完成标志
    std::atomic<int64_t> seek_time_us{0};      // seek 前的位置（用于恢复）

    // ===== HLS 相关 =====
    int64_t start_time;                         // 起始时间偏移
    std::atomic<bool> hls_is_seeking{false};   // HLS 是否正在 seek

    // ===== 回调指针 =====
    VideoPlayer* m_player;                     // 指向 VideoPlayer 对象
                                              // 用于 emit 信号
} VideoState;
```

## 三、PacketQueue — 生产者-消费者队列

### 🎬 就像传送带：demux 往上放包，video/audio 线程从上面取包

```
┌─── PacketQueue ──────────────────────────────────────┐
│                                                       │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐            │
│  │ Pkt1 │→│ Pkt2 │→│ Pkt3 │→│ Pkt4 │→│ Pkt5 │ → ... │
│  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘            │
│  ▲ first_pkt                              last_pkt ▲ │
│  │                                              │     │
│  │  ← packet_queue_get() 取出                   │     │
│  │                                              │     │
│  └── packet_queue_put() 放入 ──────────────────┘     │
│                                                       │
│  nb_packets = 5      （队列中的包数）                   │
│  size = 累计字节数    （所有包的数据大小总和）            │
│                                                       │
│  mutex + cond        （互斥锁 + 条件变量）              │
│  - put 时: lock → 入队 → cond.signal → unlock         │
│  - get 时: lock → 等待cond → 出队 → unlock            │
│                                                       │
└───────────────────────────────────────────────────────┘
```

### 3.1 packet_queue_put() — 生产者入队

```cpp
int packet_queue_put(PacketQueue *q, AVPacket *pkt)
{
    AVPacketList *pkt1 = (AVPacketList *)av_malloc(sizeof(AVPacketList));
    // 分配一个链表节点

    if (!pkt1) return -1;

    pkt1->pkt = *pkt;    // 拷贝 AVPacket 数据
    pkt1->next = nullptr; // 链表尾部，next 为空

    SDL_LockMutex(q->mutex);   // 加锁
    if (!q->last_pkt)
        q->first_pkt = pkt1;   // 队列为空，新节点就是头节点
    else
        q->last_pkt->next = pkt1;  // 队列非空，追加到尾部

    q->last_pkt = pkt1;        // 更新尾指针
    q->nb_packets++;           // 包数 +1
    q->size += pkt1->pkt.size; // 累加数据大小

    SDL_CondSignal(q->cond);   // 通知等待的消费者："有新数据了"
    SDL_UnlockMutex(q->mutex); // 解锁
    return 0;
}
```

### 3.2 packet_queue_get() — 消费者出队

```cpp
int packet_queue_get(PacketQueue *q, AVPacket *pkt, int block)
// block = 1: 阻塞等待（队列空时不返回，等到有数据）
// block = 0: 非阻塞（队列空时立即返回 -1）
{
    AVPacketList *pkt1;
    int ret;

    SDL_LockMutex(q->mutex);   // 加锁

    for (;;) {
        pkt1 = q->first_pkt;   // 取头节点
        if (pkt1) {
            // 队列非空，取出
            q->first_pkt = pkt1->next;  // 头指针后移
            if (!q->first_pkt)
                q->last_pkt = nullptr;  // 队列变空，尾指针也清空

            q->nb_packets--;       // 包数 -1
            q->size -= pkt1->pkt.size;  // 数据大小减少

            *pkt = pkt1->pkt;     // 拷贝 AVPacket 给调用者
            av_freep(&pkt1);      // 释放链表节点

            ret = 1;              // 成功取出
            break;
        } else if (!block) {
            // 队列空 + 非阻塞模式 → 立即返回
            ret = 0;
            break;
        } else {
            // 队列空 + 阻塞模式 → 等待信号
            SDL_CondWait(q->cond, q->mutex);
            // ⚠️ CondWait 会自动释放 mutex 并阻塞
            // 被 CondSignal 唤醒后自动重新获取 mutex
            // 然后继续 for 循环，再次检查队列
        }
    }

    SDL_UnlockMutex(q->mutex);   // 解锁
    return ret;
}
```

## 四、音视频同步 — 核心机制

### 🎬 就像对口型：音频是"标准"，视频要跟着音频的节奏走

### 4.1 为什么以音频为主时钟？

| 对比 | 音频中断 | 视频卡顿 |
|------|---------|---------|
| 人耳感知 | 极度敏感，几十毫秒断音就能听出 | 不太敏感，偶尔丢一帧几乎无感 |
| 时钟稳定性 | SDL 回调驱动，自然平滑递增 | 取决于解码速度，可能有波动 |
| 调整方式 | 不能丢音频（会断音） | 可以丢帧或延迟（用户几乎无感） |

**结论**：音频做主时钟，视频追音频。

### 4.2 同步逻辑（video_thread 中）

```
视频帧解码完成，得到 video_clock_us
       │
       ▼
计算差值: delay = video_clock_us - audio_clock_us
       │
       ├── delay > 阈值（视频超前音频太多）
       │     → 延迟显示（等音频追上）
       │     → SDL_Delay(delay)
       │
       ├── delay < -阈值（视频落后音频太多）
       │     → 丢帧（跳过这一帧，解码下一帧）
       │     → 不显示，继续循环
       │
       └── 差值在阈值内
             → 正常显示
             → submitFrame(QImage)
```

## 五、seek 实现

### 🎬 就像快进录像带：要告诉三个线程"跳到新位置"

### 5.1 seek 完整流程

```
用户拖动进度条 → VideoPlayer::seek(pos)
       │
       ▼
设置 seek_req = 1, seek_pos_us = pos
（通过 atomic 变量通知 demux 线程）
       │
       ▼ (demux 线程的 run() 循环检测到 seek_req)
       │
┌── demux 线程 ──┐
│  av_seek_frame(pFormatCtx, seek_pos_us, AVSEEK_FLAG_BACKWARD)
│  // AVSEEK_FLAG_BACKWARD = 定位到目标位置之前最近的关键帧
│  // 为什么？因为非关键帧不能独立解码，必须从关键帧开始
│  │
│  packet_queue_flush(audioq)  // 清空音频队列
│  packet_queue_flush(videoq)  // 清空视频队列
│  // flush 后发送 FLUSH_DATA 标记包，通知解码器刷新状态
│  │
│  audio_clock_us = 0  // 重置音频时钟 ⚠️ 很重要！
│  // 如果不重置，视频线程会认为"音频还在老位置"，持续丢帧
│  │
│  seek_req = 0  // 清除请求标志
└────────────────┘
       │
       ▼ (video_thread 和 audio_callback 检测到 FLUSH_DATA)
       │
┌── video/audio 线程 ──┐
│  检测到 FLUSH_DATA 包  │
│  → avcodec_flush_buffers()  // 刷新解码器内部缓冲区
│  → 重置解码器状态           │
│  → 继续从新位置解码         │
└─────────────────────────┘
```

### 5.2 seek 的关键细节

| 步骤 | 为什么必须做 | 不做的后果 |
|------|------------|-----------|
| `AVSEEK_FLAG_BACKWARD` | 确保定位到关键帧 | 定位到非关键帧 → 花屏 |
| `packet_queue_flush()` | 清空旧位置的包 | 播放旧数据 → 画面跳回 |
| 发送 `FLUSH_DATA` | 通知解码器刷新 | 解码器残留旧帧 → 花屏 |
| 重置 `audio_clock_us` | 避免误判 | 视频线程认为音频在老位置 → 持续丢帧 |
| `avcodec_flush_buffers()` | 清理解码器内部缓存 | 解码器输出旧帧 → 花屏 |

## 六、HLS 流式播放

### 🎬 就像看直播：不用下载完整个视频，边下边播

### 6.1 HLS 是什么？

HLS（HTTP Live Streaming）把一个完整视频切成很多小段（.ts 文件），用一个索引文件（.m3u8）记录每段的位置和时长。

```
output.m3u8（索引文件）
  ├── segment0.ts（0-10秒）
  ├── segment1.ts（10-20秒）
  ├── segment2.ts（20-30秒）
  └── ...
```

### 6.2 FFmpeg 如何播放 HLS

```cpp
// 直接用 avformat_open_input 打开 m3u8 URL
// FFmpeg 内部会自动：
//   1. 下载 m3u8 索引
//   2. 解析分片列表
//   3. 按需下载 ts 分片
//   4. 解封装成 AVPacket

avformat_open_input(&pFormatCtx, "http://10.56.5.9:8080/hls/xxx/output.m3u8", nullptr, nullptr);
```

### 6.3 HLS 的 seek

HLS 的 seek 比本地文件复杂，因为需要找到目标时间对应的 ts 分片：

```cpp
// 1. 找到目标时间对应的分片索引
int segIdx = findHlsSegmentForTime(seekPosUs);

// 2. 重建 FFmpeg 上下文，从该分片开始播放
rebuildHlsFromSegment(seekPosUs);

// 3. 内部流程：
//    - 关闭旧的 formatCtx
//    - 重新 avformat_open_input（从目标分片 URL）
//    - 重新找流信息
//    - 重新打开解码器
```

## 七、关键变量速查表

| 变量 | 类型 | 所属线程 | 读写 | 含义 |
|------|------|---------|------|------|
| `audio_clock_us` | `atomic<int64_t>` | 音频线程写，视频线程读 | 写:audio, 读:video | 音频主时钟（微秒） |
| `video_clock_us` | `atomic<int64_t>` | 视频线程读写 | 写:video, 读:video | 视频时钟（微秒） |
| `seek_req` | `atomic<int>` | UI线程写，demux线程读 | 写:UI, 读:demux | seek 请求标志 |
| `seek_pos_us` | `atomic<int64_t>` | UI线程写，demux线程读 | 写:UI, 读:demux | seek 目标位置 |
| `quit` | `atomic<bool>` | 任何线程写，所有线程读 | 写:any, 读:all | 退出标志 |
| `isPause` | `atomic<bool>` | UI线程写，所有线程读 | 写:UI, 读:all | 暂停标志 |

💡 **为什么用 `std::atomic`？** 因为这些变量在多个线程之间共享，如果用普通变量，读写可能不一致（比如读到半写的值）。`atomic` 保证读写是原子的——要么读到旧值，要么读到新值，不会读到"半新半旧"的值。

## 八、常见问题

### Q: 为什么用 SDL_Thread 而不是 QThread 做视频线程？
> 历史原因。这个播放器的代码参考了 FFmpeg 官方示例（tutorial），示例用的是 SDL_Thread。QThread 也可以，但需要额外的信号槽机制。SDL_Thread 更轻量，直接用 C 函数。

### Q: seek 时为什么会卡顿一下？
> seek 需要清空队列、刷新解码器、重新定位。这个过程中，视频线程和音频线程都在等待新数据，画面会短暂冻结。优化方向：使用关键帧索引（如有）加速定位，或异步 seek（先继续播放旧位置，seek 完成后跳转）。

### Q: 为什么不用 OpenGL 渲染视频？
> 当前用 QImage + QLabel 渲染，简单但效率低（每帧都做 YUV→RGB 转换 + 内存拷贝）。OpenGL 可以直接渲染 YUV420P，省掉 sws_scale 转换。但 OpenGL 渲染的代码复杂度高很多，当前是 MVP 阶段，先用简单方案。
