# 06 - 服务端核心模块 🖥️

> 📌 本文档详解服务端的架构设计、消息处理流程和线程安全机制。
> 服务端是整个系统的"大脑"——接收请求、处理业务、返回响应。

---

## 一、服务端架构概览

### 🎬 就像银行网点：大堂经理接客，柜员办业务

```
客户端请求进来
       │
       ▼
  EpollServer（大堂经理 —— epoll 监听连接和读取数据）
       │
       │  收到一条完整消息帧
       ▼
  ThreadPool（叫号 → 分配柜员 —— 线程池调度）
       │
       ▼
  ClientHandler::processMessage()（柜员办业务）
       │
       ├── switch(type) 路由到具体 handle 函数
       │     ├── handleRegister()    注册
       │     ├── handleLogin()       登录
       │     ├── handleUploadStart() 上传
       │     └── ... 其他 17 种操作
       │
       ▼
  sendResponse() → m_sendFunc(response) → EpollServer 异步写回
```

### 1.1 为什么用 epoll + 线程池？

| 对比项 | 单线程 Reactor | epoll + 线程池（本项目） |
|--------|---------------|------------------------|
| 网络IO | 主线程做，简单 | epoll 专做，效率高 |
| 业务处理 | 主线程做，DB查询会阻塞 | 线程池做，不阻塞IO |
| 并发能力 | 1个请求阻塞则全部等待 | 多个请求并行处理 |
| 复杂度 | 低 | 中等（需要线程安全） |

## 二、Server 类 — 服务器主控

### 2.1 关键成员变量

| 变量 | 类型 | 含义 |
|------|------|------|
| `m_threadPool` | `ThreadPool` | 线程池，处理业务逻辑 |
| `m_epollServer` | `EpollServer` | epoll IO 多路复用，处理网络读写 |
| `m_database` | `Database` | 数据库操作对象 |
| `m_clients` | `QMap<int, shared_ptr<ClientHandler>>` | 客户端映射表（fd → handler） |
| `m_hlsIp/Port/OutputDir` | `QString/int/QString` | HLS 配置（HTTP 由 Nginx 静态服务提供） |

### 2.2 启动流程

```cpp
bool Server::start(quint16 port)
{
    // 初始化数据库
    if (!m_database.init("localhost", "root", "密码", "videorecorder")) {
        return false;
    }

    // HLS 文件由 ffmpeg 写入 hls_output_dir，HTTP 分发由 Nginx 提供 /hls/

    // 启动 epoll 服务器，注册回调
    m_epollServer.setOnMessage([this](int fd, const QString &peerAddr,
                                       const QByteArray &frame) {
        // 收到完整帧 → 创建/查找 ClientHandler → 丢给线程池
        auto handler = createClientHandler(fd, peerAddr);
        m_threadPool.enqueue([handler, frame]() {
            // 在线程池中解析和处理
            quint32 type;
            QByteArray jsonData;
            // ... 从 frame 解析 type 和 jsonData
            handler->processMessage(type, jsonData);
        });
    });

    m_epollServer.start(port);
    return true;
}
```

## 三、ClientHandler 类 — 业务处理器

### 🎬 每个客户端连接对应一个 ClientHandler，就像每个客户有专属柜员

### 3.1 构造函数参数

```cpp
ClientHandler::ClientHandler(int fd, const QString &peerAddr, SendFunc sendFunc,
                             Database *db, const QString &hlsIp, int hlsPort,
                             const QString &hlsOutputDir)
```

| 参数 | 类型 | 来源 | 含义 |
|------|------|------|------|
| `fd` | `int` | epoll accept 返回 | 客户端的文件描述符（socket 编号） |
| `peerAddr` | `QString` | epoll 读取 | 客户端的 IP:Port 地址 |

| `hlsOutputDir` | `QString` | config.ini | HLS 文件输出目录 |

### 3.2 processMessage() — 消息路由总入口

```cpp
void ClientHandler::processMessage(quint32 type, const QByteArray &jsonData)
{
    QJsonParseError error;
    QJsonDocument doc = QJsonDocument::fromJson(jsonData, &error);
    // 【步骤1】把 JSON 字节流解析成 QJsonDocument
    // 如果 JSON 格式有错（如少了个引号），error 会记录错误位置

    if (error.error != QJsonParseError::NoError) {
        qDebug() << "JSON 解析失败：" << error.errorString();
        sendResponse(RESP_SERVER_ERROR, "JSON 格式错误");
        return;                    // JSON 都解析不了，直接返回错误
    }

    QJsonObject json = doc.object();   // 【步骤2】提取 JSON 对象

    switch (type) {                     // 【步骤3】根据 type 路由到不同处理函数
        case MSG_REGISTER:    handleRegister(json);    break;
        case MSG_LOGIN:       handleLogin(json);       break;
        case MSG_GET_VIDEO_LIST:    handleGetVideoList(json);    break;
        case MSG_GET_RECOMMEND_FEED: handleGetRecommendFeed(json); break;
        case MSG_SEARCH_VIDEOS:     handleSearchVideos(json);     break;
        case MSG_GET_VIDEO_INFO:    handleGetVideoInfo(json);     break;
        case MSG_INCREMENT_PLAY:    handleIncrementPlay(json);    break;
        case MSG_GET_LIKE_STATUS:   handleGetLikeStatus(json);    break;
        case MSG_TOGGLE_LIKE:       handleToggleLike(json);       break;
        case MSG_UPLOAD_START:      handleUploadStart(json);      break;
        case MSG_UPLOAD_CHUNK:      handleUploadChunk(json);      break;
        case MSG_UPLOAD_END:        handleUploadEnd(json);        break;
        case MSG_UPLOAD_RESUME:     handleUploadResume(json);     break;
        case MSG_DOWNLOAD_START:    handleDownloadStart(json);    break;
        case MSG_DOWNLOAD_CHUNK:    handleDownloadChunk(json);    break;
        case MSG_GET_MY_VIDEOS:     handleGetMyVideos(json);      break;
        case MSG_DELETE_VIDEO:      handleDeleteVideo(json);      break;
        default:
            qDebug() << "未知消息类型：" << type;
            sendResponse(RESP_SERVER_ERROR, "未知消息类型");
            break;
    }
}
```

## 四、handleRegister() — 注册处理详解

### 4.1 完整流程

```
客户端发送 MSG_REGISTER
  JSON: {"username": "Fledgling", "password": "a3f2b8c1..."}
       │
       ▼
processMessage() → switch → handleRegister()
       │
       ▼
提取 username 和 password
       │
       ▼
参数检查：是否为空？
       │
       ├── 空 → sendResponse(500, "用户名或密码不能为空", "register")
       │
       ▼ 非空
m_db->registerUser(username, password)
       │
       ├── 成功 → sendResponse(200, "注册成功", "register")
       │
       └── 失败
             ├── 用户已存在 → sendResponse(401, "用户已存在", "register")
             └── 数据库错误 → sendResponse(500, "注册失败：数据库错误", "register")
```

### 4.2 逐行解析

```cpp
void ClientHandler::handleRegister(const QJsonObject &json)
{
    QString username = json["username"].toString();
    // 从 JSON 中提取 "username" 字段
    // 如果客户端没传这个字段，toString() 返回空字符串

    QString password = json["password"].toString();
    // 从 JSON 中提取 "password" 字段
    // 这个值已经是客户端 SHA-256 加盐哈希后的 64 位十六进制字符串
    // 服务端不再做二次加密，直接存入数据库

    qDebug() << "收到注册请求：用户名 =" << username;

    if (username.isEmpty() || password.isEmpty()) {
        sendResponse(RESP_SERVER_ERROR, "用户名或密码不能为空", "register");
        // 参数校验失败，返回 500 错误
        // "register" 是 action，告诉客户端这个响应对应注册操作
        return;
    }

    if (m_db->registerUser(username, password)) {
        sendResponse(RESP_SUCCESS, "注册成功", "register");
        // 注册成功，返回 200
    } else {
        if (m_db->userExists(username)) {
            sendResponse(RESP_USER_EXISTS, "用户已存在", "register");
            // 用户名重复，返回 401
        } else {
            sendResponse(RESP_SERVER_ERROR, "注册失败：数据库错误", "register");
            // 其他数据库错误，返回 500
        }
    }
}
```

## 五、handleLogin() — 登录处理详解

### 5.1 逐行解析

```cpp
void ClientHandler::handleLogin(const QJsonObject &json)
{
    QString username = json["username"].toString();
    QString password = json["password"].toString();
    // password 是客户端发来的 SHA-256 哈希值

    if (username.isEmpty() || password.isEmpty()) {
        sendResponse(RESP_SERVER_ERROR, "用户名或密码不能为空", "login");
        return;
    }

    if (m_db->loginUser(username, password)) {
        // 登录成功！
        m_username = username;
        // 记录当前连接的用户名（后续操作可能需要）

        // 生成 token
        QString tokenRaw = username + "_"
            + QString::number(QDateTime::currentMSecsSinceEpoch()) + "_"
            + QString::number(qrand());
        // tokenRaw = "Fledgling_1686xxx_12345"
        // 由 用户名 + 时间戳 + 随机数 组成，保证唯一性

        QByteArray tokenHash = QCryptographicHash::hash(
            tokenRaw.toUtf8(), QCryptographicHash::Sha256);
        // 对 tokenRaw 做 SHA-256 哈希，得到 32 字节的二进制值

        QString token = QString(tokenHash.toHex());
        // 转成 64 位十六进制字符串
        // 如 "e8f3a1b2c4d5..."

        QJsonObject data;
        data["token"] = token;        // 会话凭证
        data["username"] = username;   // 用户名

        sendResponse(RESP_SUCCESS, "登录成功", "login", data);
        // 返回 200 + token + 用户名

    } else {
        // 登录失败，区分"用户不存在"和"密码错误"
        if (!m_db->userExists(username)) {
            sendResponse(RESP_USER_NOT_FOUND, "用户不存在", "login");
            // 返回 404
        } else {
            sendResponse(RESP_PASSWORD_ERROR, "密码错误", "login");
            // 返回 402
        }
    }
}
```

### 5.2 Token 是什么？

💡 **用大白话说**：Token 就像酒店房卡。你用身份证（密码）办入住，前台给你一张房卡（token），之后用房卡而不是身份证来进出。

**Token 生成算法**：
```
原始串 = "用户名_毫秒时间戳_随机数"
Token = SHA-256(原始串) 的十六进制表示
```

**为什么这样设计？**
- 用户名保证不同用户的 token 不同
- 时间戳保证同一用户每次登录 token 不同
- 随机数防止毫秒级碰撞
- SHA-256 单向哈希，不能从 token 反推出用户信息

## 六、sendResponse() — 响应发送

### 6.1 两种重载

```cpp
// 重载1：2 参数（简化版，内部转发到 4 参数）
void sendResponse(int code, const QString &message)
{ sendResponse(code, message, QString(), QJsonObject()); }

// 重载2：4 参数（完整版）
void sendResponse(int code, const QString &message,
                  const QString &action, const QJsonObject &data = QJsonObject());
```

💡 **为什么不保留 3 参数重载？** 因为之前 3 参数 `sendResponse(int, QString, QJsonObject)` 和 4 参数 `sendResponse(int, QString, QString, QJsonObject)` 会导致重载歧义——`const char*` 字面量如 `"register"` 可能被隐式转成 `QJsonObject`。删掉 3 参数版本彻底消除歧义。

### 6.2 4 参数版本逐行解析

```cpp
void ClientHandler::sendResponse(int code, const QString &message,
                                  const QString &action, const QJsonObject &data)
{
    // 【步骤1】构造 JSON 响应对象
    QJsonObject json;
    json["code"] = code;              // 状态码：200/401/402/404/500
    json["message"] = message;        // 消息文本："登录成功"、"密码错误" 等
    if (!action.isEmpty()) {
        json["action"] = action;      // 动作标识（非空时才添加）
    }                                 // 如 "login"、"register"、"video_list"
    if (!data.isEmpty()) {
        json["data"] = data;          // 附加数据（非空时才添加）
    }                                 // 如 {"token": "xxx", "username": "xxx"}

    QByteArray jsonData = QJsonDocument(json).toJson(QJsonDocument::Compact);
    // 序列化成紧凑 JSON 字符串

    // 【步骤2】构造二进制帧
    QByteArray response;

    quint32 type = qToBigEndian(static_cast<quint32>(MSG_RESPONSE));
    // MSG_RESPONSE = 3，转成大端序
    response.append(reinterpret_cast<const char*>(&type), 4);
    // 写入 4 字节 type

    quint32 length = qToBigEndian(static_cast<quint32>(jsonData.size()));
    // JSON 数据长度，转成大端序
    response.append(reinterpret_cast<const char*>(&length), 4);
    // 写入 4 字节 length

    response.append(jsonData);
    // 写入 JSON 数据体

    // 【步骤3】通过回调函数发送
    if (m_sendFunc)
        m_sendFunc(response);
    // m_sendFunc 是 EpollServer 提供的发送回调
    // 把数据交给 epoll 的写缓冲区异步发送
    // 不是直接 write()，因为当前可能在线程池中执行
    // EpollServer 内部保证线程安全的异步写入
}
```

## 七、线程安全设计

### 🎬 就像厨房：多个厨师共用一个冰箱，必须有人管理取放顺序

### 7.1 三层安全保障

```
┌─────────────────────────────────────────────────────┐
│ 第一层：网络IO隔离                                    │
│   所有 socket 读写都在 epoll 线程                     │
│   工作线程不直接操作 socket                           │
│   通过 m_sendFunc 回调，由 epoll 线程异步写入          │
├─────────────────────────────────────────────────────┤
│ 第二层：数据库互斥                                    │
│   Database 所有操作由 QMutex 保护                     │
│   QMutexLocker 自动加锁/解锁（RAII 模式）             │
│   同一时刻只有一个线程能操作数据库                      │
├─────────────────────────────────────────────────────┤
│ 第三层：连接生命周期管理                               │
│   ClientHandler 用 shared_ptr 管理                   │
│   客户端断开时从 m_clients 移除                       │
│   定时清理（m_cleanupTimer）回收断开的连接              │
└─────────────────────────────────────────────────────┘
```

### 7.2 为什么不用 QMetaObject::invokeMethod？

之前用 `invokeMethod` + lambda 实现跨线程发送响应，但在 Qt 5.9 上编译失败（`QPointer` 模板推导问题）。现在改用 `m_sendFunc` 回调函数，由 `EpollServer` 负责线程安全的异步写入。

## 八、服务端消息处理全流程

```
客户端发来数据
       │
       ▼
EpollServer::onReadable()
       │  read() 读取字节流
       │  拼帧：检查是否凑齐 [4B type][4B length][length 字节 JSON]
       │
       ▼ 完整帧就绪
回调 Server 的 onMessage(fd, peerAddr, frame)
       │
       ▼
Server::createClientHandler(fd, peerAddr)
       │  创建或查找 ClientHandler
       │  封装 sendFunc 回调
       │
       ▼
ThreadPool::enqueue([handler, frame]() {
    handler->processMessage(type, jsonData);
})
       │  把任务丢给线程池
       │  不阻塞 epoll 线程
       │
       ▼ (线程池中的工作线程执行)
ClientHandler::processMessage(type, jsonData)
       │  JSON 解析
       │  switch(type) 路由
       │
       ▼
handleXxx(json)
       │  业务处理
       │  可能调用 Database（QMutex 保护）
       │
       ▼
sendResponse(code, message, action, data)
       │  构造 JSON + 二进制帧
       │
       ▼
m_sendFunc(response)
       │  通过 EpollServer 异步写回客户端
       │
       ▼
EpollServer::onWritable() → write() 发送数据
```

## 九、常见问题

### Q: 为什么 ClientHandler 不是 QObject？
> 因为它在线程池中使用，不依赖 Qt 的信号槽机制。发送响应通过函数回调 `m_sendFunc`，不需要 Qt 的事件循环。

### Q: Database 的 QMutex 会不会成为瓶颈？
> 在高并发场景下会。当前所有 DB 操作串行执行，同一时刻只能有一个查询。改进方向：读写分离（读操作用多个连接，写操作保持单连接），或者用连接池。

### Q: 为什么 registerUser 里不能调用 userExists？
> 因为 `registerUser` 已经持有 `m_mutex` 锁，`userExists` 也会尝试加锁。`QMutex` 默认是非递归的，同一线程第二次加锁会**死锁**（永久阻塞）。所以 `registerUser` 内部直接写了 SQL 查询来检查用户是否存在。
