### 3.2 恒定时间比较 — constantTimeEqualUtf8()

💡 **为什么不用 `==` 比较？** 因为 `==` 是短路比较——发现第一个不同的字符就立即返回 `false`。攻击者可以通过**测量响应时间**来逐字符猜测密码（时序攻击）。

比如：
- 正确密码：`a3f2b8c1d4e5...`
- 攻击者尝试：`000000000000...` → 第 1 个字符就不同 → 0.001ms 返回
- 攻击者尝试：`a00000000000...` → 前 1 个字符相同 → 0.002ms 返回
- 攻击者尝试：`a30000000000...` → 前 2 个字符相同 → 0.003ms 返回

通过时间差异，攻击者可以逐字符猜出密码！

**恒定时间比较**的原理：**无论在哪一位发现不同，都比较完所有字符**，这样响应时间永远一样。

```cpp
bool constantTimeEqualUtf8(const QByteArray &a, const QByteArray &b)
{
    const int maxLen = qMax(a.size(), b.size());
    // 取两个字符串中较长的长度

    volatile char diff = 0;
    // ⚠️ volatile 关键字很重要！
    // 告诉编译器：不要优化这个变量
    // 没有 volatile，编译器可能会优化掉循环（因为它发现可以提前返回）
    // 有了 volatile，编译器必须逐次读取和写入 diff

    for (int i = 0; i < maxLen; ++i) {
        // 遍历到最长的长度，短的用 0 填充

        const char ca = (i < a.size()) ? a[i] : 0;
        // 如果 a 还有字符，取 a[i]；否则取 0

        const char cb = (i < b.size()) ? b[i] : 0;
        // 如果 b 还有字符，取 b[i]；否则取 0

        diff |= static_cast<char>(ca ^ cb);
        // XOR 运算：相同字符 XOR 结果为 0，不同字符 XOR 结果非 0
        // |= 操作：把每一位的 diff 累积起来
        // 只要有一个字符不同，diff 就不为 0
        // ⚠️ 不管在第几位发现不同，循环都不会 break，继续比较到末尾
    }

    return diff == 0 && a.size() == b.size();
    // diff == 0：所有字符都相同
    // a.size() == b.size()：长度也相同
    // 两个条件都满足才算相等
}
```

### 3.3 兼容旧密码 — verifyStoredPassword()

💡 **问题**：如果数据库里有旧版用户，密码是明文存储的（如 `"123456"`），而新版客户端发来的是哈希值，直接比较肯定不匹配。

```cpp
bool verifyStoredPassword(const QString &stored, const QString &clientSent)
{
    const QString storedTrim = stored.trimmed();
    const QString clientTrim = clientSent.trimmed();
    const QByteArray storedBytes = normalizeHexUtf8(storedTrim.toUtf8());
    QByteArray clientBytes = normalizeHexUtf8(clientTrim.toUtf8());
    // normalizeHexUtf8：如果是 64 位十六进制，统一转小写
    // 防止大小写不一致导致比较失败（如 "A3F2" vs "a3f2"）

    // 【情况1】两边都是哈希，直接比较
    if (constantTimeEqualUtf8(storedBytes, clientBytes))
        return true;

    // 【情况2】数据库存的是明文（旧数据）
    if (!isSha256Hex(storedTrim)) {
        // stored 不是 64 位十六进制 → 可能是明文
        const QString legacyHash = authPasswordHash(storedTrim);
        // 把明文密码也哈希一下
        if (constantTimeEqualUtf8(normalizeHexUtf8(legacyHash.toUtf8()), clientBytes)) {
            // 哈希后匹配 → 允许登录
            qDebug() << "明文密码已匹配（建议重新注册升级存储）";
            return true;
        }
    }

    // 【情况3】客户端发来明文（旧版客户端）
    if (!isSha256Hex(clientTrim)) {
        const QString fromPlain = authPasswordHash(clientTrim);
        if (constantTimeEqualUtf8(storedBytes, normalizeHexUtf8(fromPlain.toUtf8())))
            return true;
    }

    return false;  // 都不匹配 → 密码错误
}
```

### 3.4 密码自动升级

```cpp
// 在 loginUser() 中，如果验证通过但数据库存的是明文，自动升级
if (verifyStoredPassword(dbPassword, password)) {
    if (!isSha256Hex(dbPassword)) {
        // 数据库存的是明文，升级为哈希
        QSqlQuery up;
        up.prepare("UPDATE users SET password = ? WHERE username = ?");
        up.addBindValue(password.trimmed());  // 存入客户端发来的哈希值
        up.addBindValue(username);
        if (up.exec())
            qDebug() << "已将密码存储升级为哈希";
    }
    return true;
}
```

## 四、SQL 注入防护

### 🎬 就像保安检查包裹：不能用字符串拼接，要用参数化查询

### ❌ 危险写法（字符串拼接）

```cpp
// ⚠️ 千万不要这样写！
QString sql = "SELECT * FROM users WHERE username = '" + username + "'";
query.exec(sql);

// 如果 username = "admin'; DROP TABLE users; --"
// 最终 SQL = "SELECT * FROM users WHERE username = 'admin'; DROP TABLE users; --'"
// 表被删了！这就是 SQL 注入攻击
```

### ✅ 安全写法（参数化查询）

```cpp
// ✅ 本项目的写法
QSqlQuery query;
query.prepare("SELECT * FROM users WHERE username = ?");
// ? 是占位符，数据库驱动会安全地替换

query.addBindValue(username);
// addBindValue 按顺序替换 ?
// 数据库驱动会自动处理特殊字符，不会被注入

query.exec();
```

**为什么参数化查询安全？** 因为 `?` 占位符的值由数据库驱动**参数化传递**，不会拼进 SQL 语句。即使 `username` 里包含 `'` 或 `;`，也不会影响 SQL 结构。

## 五、QMutex 互斥锁

### 🎬 就像公共厕所的门锁：同一时刻只能一个人用

```cpp

{
    QMutexLocker locker(&m_mutex);
    // 【说明】QMutexLocker 是 RAII 模式的锁守卫

    // 即使函数中途 return 或抛异常，锁都会被正确释放
    //
    // 等价于：
    //   m_mutex.lock();
    //   ... 函数体 ...
    //   m_mutex.unlock();
    // 但更安全 —— 不会忘记 unlock

    // ... 数据库操作 ...

    // locker 在函数结束时自动析构，释放锁
}
```

### 5.1 为什么 Database 需要锁？

因为多个工作线程可能同时调用 Database 的方法。如果两个线程同时执行 `INSERT INTO users`，可能导致数据竞争。

### 5.2 死锁陷阱

⚠️ **`registerUser` 里不能调用 `userExists`！**

```
线程A: registerUser()
  → lock(m_mutex)      ← 加锁成功
    → userExists()
      → lock(m_mutex)  ← ❌ 死锁！m_mutex 已经被当前线程持有
                         QMutex 默认非递归，同一线程重复加锁会永久阻塞
```

**解决方案**：在 `registerUser` 内部直接写 SQL 检查用户是否存在，不调用 `userExists`。

## 六、点赞功能 — toggleVideoLike()

### 6.1 点赞/取消点赞的切换逻辑

```cpp
QVariantMap Database::toggleVideoLike(int videoId, int userId)
{
    QMutexLocker locker(&m_mutex);

    // 【步骤1】开启事务
    if (!m_db.transaction()) {
        // 事务 = 把多个 SQL 操作包装成原子操作
        // 要么全部成功，要么全部回滚
        return result;
    }

    // 【步骤2】检查是否已点赞
    QSqlQuery checkQuery;
    checkQuery.prepare("SELECT 1 FROM video_likes WHERE user_id = ? AND video_id = ? LIMIT 1");
    checkQuery.addBindValue(userId);
    checkQuery.addBindValue(videoId);
    checkQuery.exec();

    const bool alreadyLiked = checkQuery.next();
    // next() 返回 true = 查到了记录 = 已经点过赞

    // 【步骤3】根据状态执行不同操作
    if (alreadyLiked) {
        // 取消点赞
        // DELETE FROM video_likes
        // UPDATE videos SET likes_count = GREATEST(likes_count - 1, 0)
        // GREATEST 防止 likes_count 变成负数
        result["liked"] = false;
    } else {
        // 添加点赞
        // INSERT INTO video_likes
        // UPDATE videos SET likes_count = likes_count + 1
        result["liked"] = true;
    }

    // 【步骤4】查询最新点赞数
    QSqlQuery countQuery;
    countQuery.prepare("SELECT likes_count FROM videos WHERE id = ?");
    countQuery.addBindValue(videoId);
    countQuery.exec();
    // ...

    // 【步骤5】提交事务
    if (!m_db.commit()) {
        m_db.rollback();  // 提交失败，回滚所有操作
        return result;
    }

    result["success"] = true;
    result["likesCount"] = countQuery.value(0).toInt();
    return result;
}
```

### 6.2 为什么用事务？

点赞涉及两张表的操作：`video_likes` 和 `videos`。如果第一步成功但第二步失败，数据就不一致了。事务保证两步要么都成功，要么都失败。

## 八、搜索功能 — searchVideos()

### 8.1 LIKE 查询 + 防注入

```cpp
QVariantList Database::searchVideos(const QString &keyword, int offset, int limit)
{
    QString kw = keyword.trimmed();
    // 去首尾空格

    kw.replace(QLatin1Char('%'), QString());
    kw.replace(QLatin1Char('_'), QString());
    // ⚠️ 删除 SQL LIKE 通配符！
    // % 和 _ 在 LIKE 中有特殊含义
    // 如果用户输入 "%"，不加处理会匹配所有记录
    // 删除它们后，用户只能做精确的文字搜索

    if (kw.isEmpty())
        return list;  // 空关键词不搜索

    const QString pattern = QLatin1Char('%') + kw + QLatin1Char('%');
    // pattern = "%关键词%"
    // % 在前后表示：匹配包含关键词的任意字符串

    QSqlQuery query;
    query.prepare(
        "SELECT ... FROM videos v "
        "LEFT JOIN users u ON v.user_id = u.id "
        "WHERE v.title LIKE ? OR u.username LIKE ? OR v.description LIKE ? "
        "ORDER BY v.created_at DESC LIMIT ? OFFSET ?"
    );
    // 在标题、用户名、描述三个字段中搜索
    // 三个 ? 都用同一个 pattern

    query.addBindValue(pattern);  // WHERE v.title LIKE ?
    query.addBindValue(pattern);  // OR u.username LIKE ?
    query.addBindValue(pattern);  // OR v.description LIKE ?
    query.addBindValue(limit);
    query.addBindValue(offset);
}
```

## 九、热度排序 — getHotVideoList()

### 9.1 SQL 中的热度公式

```sql
hot_score = (play_count + α * likes_count + β * comments_count)
            / POW(GREATEST(TIMESTAMPDIFF(HOUR, created_at, NOW()), 0) + T, γ)
```

💡 **用大白话说**：
- 分子 = 互动量（播放+点赞+评论），越多越热
- 分母 = 时间衰减，越老越冷
- α、β 是互动权重：1 个点赞 ≈ α 个播放，1 条评论 ≈ β 个播放
- T 是时间偏移：防止新视频（0 小时）分母为 0 或接近 0 导致分数爆炸
- γ 是衰减指数：越大衰减越快

### 9.2 代码中的参数

```cpp
query.addBindValue(alpha);   // α = 3.0（1个点赞等于3个播放）
query.addBindValue(beta);    // β = 2.0（1条评论等于2个播放）
query.addBindValue(T);       // T = 2.0（时间偏移，防止除零）
query.addBindValue(gamma);   // γ = 1.5（衰减指数）
```
